# Operator guide

Runbook for the human running the World Cup 2026 Pool in production.
For the one-time bootstrap walkthrough see the
[Quick start](../README.md#local-development) and
[First-time setup](../README.md#first-time-setup-operator-runbook) sections in
[`README.md`](../README.md). This document expands on the day-to-day pieces
the README intentionally leaves short.

## Standing up a new environment

For a fresh staging or production environment, follow the README's
[First-time setup](../README.md#first-time-setup-operator-runbook) in order,
then come back here for the variations below.

### Seeding a fresh fixture set

`supabase/seed/matches.sql` is the canonical fixture list (104 matches,
generated by `scripts/generate-fixtures-sql.mjs`). Apply it after the schema
migration:

```bash
psql "$DATABASE_URL" -f supabase/seed/matches.sql
# or paste into the Supabase SQL editor
```

The file ends with an assertion that exactly 104 rows are present:

```sql
do $$
begin
  if (select count(*) from public.matches) <> 104 then
    raise exception 'expected 104 matches';
  end if;
end$$;
```

If you need to re-seed (e.g. a staging project), truncate first — the
`on delete cascade` chain wipes predictions and scores too:

```sql
truncate public.matches cascade;
```

### Promoting the first admin

The README covers this for the owner. For any **subsequent** admin:

1. Have the new admin sign in once via the deployed app so a row exists in
   `auth.users` and `public.profiles`.
2. In the Supabase SQL editor (which runs as `service_role`, so the
   self-promotion guard at
   [`supabase/migrations/20260513000000_init.sql:297`](../supabase/migrations/20260513000000_init.sql)
   allows it):

   ```sql
   update public.profiles
      set is_admin = true
    where id = (select id from auth.users where email = 'new-admin@example.com');
   ```

3. The "Admin" nav item appears on their next request.

Demoting is the same with `is_admin = false`.

## Daily operations during the tournament

### Add or fix a fixture

Admin → "New fixture" form on `/admin/matches`. The server action calls the
service-role client and validates with zod
([`app/(admin)/admin/matches/actions.ts:53`](<../app/(admin)/admin/matches/actions.ts>)):

- `stage` must be one of `group | r32 | r16 | qf | sf | third | final`.
- `group_code` must match `^[A-La-l]$` (uppercased on save) or be empty.
- `home_team` and `away_team` are non-empty and must differ.
- New fixtures require a future `kickoff_at` ISO timestamp.

Edits to existing fixtures are allowed even after kickoff (in case the schedule
changes mid-tournament), but `kickoff_at` is the lock boundary for that match —
move it forward to give users more time, or back to lock earlier.

### Enter a final score

On `/admin/matches`, locate the fixture, type the goals into the inline
result form, set status to `final`, and press **Save result**.

What happens behind the scenes:

1. `setMatchResult` server action calls the admin client to `UPDATE matches`
   ([`app/(admin)/admin/matches/actions.ts:85`](<../app/(admin)/admin/matches/actions.ts>)).
2. Postgres trigger `trg_recompute_scores_on_match_change` fires
   `compute_match_scores(match_id)` inside the same transaction.
3. The server action calls `revalidateTag('leaderboard', 'max')` so the next
   leaderboard request sees the new standings.

### Correct a previously entered result

Edit the score on the admin page and save again. The trigger deletes the old
`scores` rows for that match and inserts new ones — see
["Recompute fires on score correction"](../openspec/changes/add-world-cup-2026-pool/specs/scoring/spec.md)
in the scoring spec.

### Cancel or void a match

Set `status` to `cancelled`, or clear `home_score` / `away_score`. The
recompute removes that match's rows from `scores`, so it stops contributing
to the leaderboard.

### Force recompute

The "Force recompute scores" button on each fixture row calls
`compute_match_scores` via RPC
([`app/(admin)/admin/matches/actions.ts:114`](<../app/(admin)/admin/matches/actions.ts>)).
Use it after any out-of-band write — for example:

- You hand-edited `scores` in the SQL editor.
- You bulk-imported predictions outside the app.
- You disabled the trigger for a maintenance window.

The recompute is idempotent, so it is safe to run repeatedly.

## Monitoring queries

Run these in the Supabase SQL editor as needed.

```sql
-- Match status counts (sanity check after seeding)
select status, count(*) from public.matches group by status;

-- Predictions per user (top 20)
select pr.display_name, count(*) as picks
  from public.predictions p
  join public.profiles pr on pr.id = p.user_id
 group by pr.display_name
 order by picks desc
 limit 20;

-- Final matches still missing scores (should be empty during the tournament)
select id, home_team, away_team, kickoff_at, status, home_score, away_score
  from public.matches
 where status = 'final' and (home_score is null or away_score is null);

-- Verify recompute parity for a single match (expect counts to match)
select count(*) as predictions
  from public.predictions where match_id = '<MATCH_UUID>';
select count(*) as scores
  from public.scores where match_id = '<MATCH_UUID>';

-- Inspect a user's run
select * from public.v_leaderboard_overall where user_id = '<USER_UUID>';
```

## Common errors and remedies

### "Predictions are locked — kickoff has passed."

User-facing string returned by `submitPrediction` when Postgres rejects the
write with code `42501` (RLS violation) — typically because the match's
`kickoff_at <= now()`. Nothing to do: this is the lock working as intended.
If the kickoff time itself is wrong, fix it on the admin page; the RLS check
re-evaluates on the next attempt.

### "Admin only" (thrown from server actions)

`assertAdmin()` ran without a signed-in user, or the user's profile has
`is_admin = false`. Confirm `select is_admin from public.profiles where id
= …` and re-promote if necessary.

### "Home and away teams must differ"

Hit when an admin submits identical teams on the fixture form. Fix the input
and re-submit.

### "Kickoff must be in the future for new fixtures"

Insert-time guard in the server action
([`app/(admin)/admin/matches/actions.ts:74`](<../app/(admin)/admin/matches/actions.ts>)).
Edits to existing fixtures bypass this — only `INSERT` requires a future
kickoff.

### A user's score is wrong on the leaderboard

1. Confirm the match has `status = 'final'` and both scores set.
2. Check `public.scores` for the `(user_id, match_id)` row.
3. If the row is missing or stale, click "Force recompute scores" on the
   admin page. The leaderboard cache is invalidated automatically.

### "Missing required environment variable: …"

`lib/env.ts` fails fast at module load. The missing variable name is in the
message. Set it in Vercel project settings (Production and Preview) and
redeploy. Locally, set it in `.env.local`.

### "Production email sender is misconfigured: …"

The email dispatchers (result emails, prediction reminders, quiz reminders)
run a guard at the start of every run
([`lib/notifications/email-sender-config.ts`](../lib/notifications/email-sender-config.ts)).
In **production** (`NODE_ENV === "production"`) it emits a single
`console.warn` and sets `senderMisconfigured: true` on the dispatch summary
(visible in the cron logs and the operations control room) when either:

- **`RESEND_API_KEY` is unset** — the dispatcher no-ops and sends nothing.
  Result: reminders and result emails silently stop going out.
- **`EMAIL_FROM` still resolves to the Resend sandbox sender**
  (`World Cup Pools <onboarding@resend.dev>`, the dev-only default) — emails
  still send, but from a shared sandbox address that hurts deliverability and
  brand trust and is likely to land in spam.

**Required production configuration** (Vercel project settings → Production):

- `RESEND_API_KEY` — your Resend API key.
- `EMAIL_FROM` — a sender on a **Resend verified domain**, e.g.
  `World Cup Pools <noreply@your-verified-domain.com>`. The display name is
  cosmetic; the address inside `<…>` must be on a domain verified in Resend.

The guard only detects the two known defaults above; verifying the domain in
Resend remains an ops responsibility. The warning is non-fatal — dispatch
never throws or fails the cron because of it.

## Redeploy and rollback

### Redeploy

Vercel auto-deploys on push to the connected branch (`main` in production).
To trigger a redeploy without a code change:

- Vercel dashboard → Deployments → "Redeploy" on the latest commit, or
- `git commit --allow-empty -m "chore: redeploy" && git push`.

### Rollback

1. Vercel dashboard → Deployments.
2. Find the last known-good deployment.
3. "Promote to production" (or "Redeploy" → "use existing build cache").

The database is **not** rolled back by a Vercel rollback. If a schema change
is part of the bad deploy, prepare a corrective forward migration. There is
no down-migration policy for this project — Supabase migrations are
forward-only.

### Hot-fixing data

For one-off data fixes, use the Supabase SQL editor (runs as `service_role`,
which bypasses RLS but still trips the self-promotion guard on `profiles`).
Always wrap multi-statement edits in a transaction:

```sql
begin;
update public.matches set kickoff_at = '2026-06-12T20:00:00Z' where id = '...';
-- verify
commit;
```

## Operational notes

- The middleware refreshes Supabase session cookies on every non-static
  request ([`middleware.ts:5`](../middleware.ts)). If a user reports being
  signed out mid-session, check the Vercel function logs for `middleware`
  errors.
- The matches list and match detail pages render dynamically — every request
  hits Postgres. This is fine at the expected scale (low thousands of users),
  but if traffic spikes, swap in `'use cache'` + `cacheTag('matches')` on
  `app/(public)/matches/page.tsx`. The admin actions already call
  `revalidateTag('leaderboard', 'max')`; add a parallel `'matches'` tag if
  you cache.
- The site URL used by Supabase Auth redirects is resolved by
  `lib/env.ts:resolveSiteUrl()` in this order: `NEXT_PUBLIC_SITE_URL` →
  `VERCEL_PROJECT_PRODUCTION_URL` → `VERCEL_URL` → `http://localhost:3000`.
  Override `NEXT_PUBLIC_SITE_URL` when you need to point preview deploys at
  a custom domain.
