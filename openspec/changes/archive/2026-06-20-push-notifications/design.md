## Context

The app's entire re-engagement layer is email-only and mature. Each dispatcher (`lib/notifications/prediction-reminder-emails.ts`, `result-emails.ts`, `quiz-reminder-emails.ts`, the digests, `comeback-emails.ts`) follows the same shape: a service-role admin client, a per-player opt-in read from `profiles.email_prefs` via `isOptedIn` (`lib/email-prefs.ts`), recipient emails resolved from `auth.users`, a Resend `batch.send` (≤100/batch), and a **ledger row written only after the provider accepts** — giving at-most-once delivery that survives crashes and idempotent re-runs. Crons (`app/api/cron/*`) authenticate with Bearer `CRON_SECRET`, wrap work in `recordRun(...)`, and isolate any dispatch failure into a zero-summary `200` so a flaky run never trips Vercel's cron alerting.

The two push triggers correspond to audiences the app **already computes**:

- **Match-needed.** `computePendingPredictionReminders` (pure, exported, unit-tested in `prediction-reminder-emails.ts`) yields, per opted-in player, today's confirmed still-open matches they have not predicted, after `filterRecipientsAtLocalHour` buckets recipients to ~7am local. The prediction-reminders cron (`app/api/cron/prediction-reminders/route.ts`, hourly) drives it.
- **Standing-changed.** `captureRankSnapshot` (`lib/notifications/rank-snapshot.ts`) records every player's overall rank from `v_leaderboard_overall` **before** `runSync()` recomputes scores; after the recompute, `result-emails.ts` derives each affected player's movement with `computeRankDelta(previousRank, newRank)` and renders "you moved up N to #X". This is driven from the `sync-matches` cron (`app/api/cron/sync-matches/route.ts`), already wired as an isolated step after `runSync()`.

What is missing is purely the **delivery channel**. The cleanest design reuses both audience computations verbatim and adds Web Push as a second send alongside the email send in each path, gated by a new `push` preference key that the existing `email_prefs` UI already iterates.

Web Push needs three pieces the app does not have: a registered **service worker** (to receive `push` events while the tab is closed), **VAPID** keys (to authenticate the application server to the push service), and a **`push_subscriptions`** store (one row per browser endpoint, written by a client subscribe flow). These are the bulk of the work and are isolated into phase 1; the trigger wiring in phases 2–3 is small.

## Goals / Non-Goals

**Goals:**
- Deliver an opt-in Web Push for "a match needs your pick today" and "your standing changed", reusing the existing prediction-reminder pending set and the existing result-email rank delta — no re-derivation of audience.
- Send push **alongside** the existing email crons (same schedules, same `recordRun`/isolation discipline), not on new infrastructure.
- At-most-once push per trigger, surviving crashes and idempotent re-runs, via a push-specific ledger mirroring the email ledgers.
- Honor a single per-player `push` opt-in (new `email_prefs` key, default opted-IN) so the existing account-menu toggles and the unsubscribe surfaces control push with no new preference plumbing.
- No-op cleanly (no crash, no send) when VAPID env is unset, matching the `RESEND_API_KEY`-unset posture of the email dispatchers.
- Prune dead subscriptions (`410 Gone` / `404`) so the store stays healthy without a separate cron.

**Non-Goals:**
- Native mobile apps, APNs, or FCM — Web Push (VAPID) only.
- An in-app notification center, history, or unread badge.
- Per-timezone push scheduling beyond the local-7am bucketing the prediction reminder already applies.
- Push for quiz reminders, digests, or comeback emails — only the two analysis-named triggers.
- A separate push preference per trigger, rich-media/image payloads, or action buttons beyond a single click-through deep link.
- Any change to scoring, ranks, the leaderboard view, or competition data.

## Decisions

- **Web Push with VAPID via the `web-push` library.** Standard, dependency-light, works on Chrome/Edge/Firefox desktop+Android and on iOS Safari 16.4+ when installed to the Home Screen. VAPID keys are generated once (`npx web-push generate-vapid-keys`) and stored as env: `NEXT_PUBLIC_VAPID_PUBLIC_KEY` (client, to call `pushManager.subscribe`), `VAPID_PRIVATE_KEY` + `VAPID_SUBJECT` (server, to sign sends). All three nullable in `lib/env.ts`; every send path no-ops when any is unset — same dormant-until-configured posture as `resendApiKey`/`footballDataToken`/`openrouterApiKey`.
- **One `push_subscriptions` row per browser endpoint, RLS owner-only.** New table `public.push_subscriptions (id uuid pk default gen_random_uuid(), user_id uuid not null references public.profiles(id) on delete cascade, endpoint text not null unique, p256dh text not null, auth text not null, user_agent text, created_at timestamptz not null default now(), last_seen_at timestamptz not null default now(), failure_count int not null default 0)`. RLS enabled with policies allowing a signed-in user to `select`/`insert`/`delete` only rows where `user_id = auth.uid()`; the cron send paths read across all users via the **service-role admin client** (bypasses RLS, like every other dispatcher). The unique `endpoint` lets the subscribe action upsert idempotently when the same browser re-subscribes. DB migration required.
- **Subscribe UI lives in the existing prefs panel (`components/user-menu.tsx`).** That component already renders the `email_prefs` toggles by iterating `EMAIL_PREF_KEYS` and calls `updateEmailPrefs` (`app/[locale]/profile-actions.ts`). Adding `push` to `EMAIL_PREF_KEYS` makes it appear as a toggle automatically. The toggle's `onCheckedChange`, when turning on, additionally runs the browser subscribe flow (register `/sw.js`, `Notification.requestPermission()`, `pushManager.subscribe({ userVisibleOnly: true, applicationServerKey })`, then persist via a new `savePushSubscription` server action); turning off (or denied permission) removes the subscription via `removePushSubscription` and sets the pref false. This keeps push opt-in co-located with the email opt-ins and reuses the merged-prefs save path.
- **Reuse the `email_prefs.push` key as the single opt-in gate.** Rather than a new column or a per-trigger matrix, `push` joins `prediction_reminder`/`result`/`quiz_reminder`/`results_digest`/`recap_digest`/`comeback` in `email_prefs`, default opted-IN, read with the existing `isOptedIn`. A player with `email_prefs.push = false` (or no permission / no subscription) gets no push; a player with no key/null/non-boolean is treated as opted-in (but still sends nothing without a stored subscription, so default-on is harmless). The footer/account unsubscribe surfaces that iterate `EMAIL_PREF_KEYS` pick it up for free.
- **A reusable sender that prunes dead endpoints.** `lib/notifications/web-push.ts` exposes `sendWebPush(subscription, payload)` over the `web-push` library, configured once with the VAPID keys. It returns a result distinguishing success from a `410 Gone`/`404` "subscription expired" so the caller deletes that `push_subscriptions` row (or increments `failure_count` and prunes past a threshold). No-ops (returns a skip) when VAPID env is unset. The payload is a small JSON: `{ title, body, url, tag }`; the service worker reads it in its `push` handler.
- **Service worker `public/sw.js`, served from origin root.** Minimal: a `push` listener that `event.waitUntil(self.registration.showNotification(title, { body, data: { url }, tag }))` and a `notificationclick` listener that focuses an existing client on `url` or opens it. Served as a static asset (scope `/`) so it controls the whole origin; no Next.js route needed. Registered lazily from the subscribe flow, not on every page load.
- **Phase 2 — match-needed push rides the prediction-reminders cron.** After the email send in `dispatchPredictionReminders` (or a sibling that shares `computePendingPredictionReminders` + `filterRecipientsAtLocalHour`), for each pending player who is `push`-opted-in and has subscriptions, send one push: title/body from the `pushNotifications` namespace ("{n} matches need your pick today"), `url = /matches?picks=needed`. Idempotent per player per UTC day via a push ledger so the hourly cron never double-pushes — reuse `prediction_reminder_log` with a `pushed_at`/`channel` discriminator, or a sibling `prediction_reminder_push_log`. The push step is isolated: its failure is logged and never affects the email summary.
- **Phase 3 — standing-changed push rides the sync-matches cron.** Inside the existing isolated result-email step (or immediately after), for each affected player who is `push`-opted-in and has subscriptions, send one push using the already-computed `RankDelta` + new rank: title/body from `pushNotifications` ("You moved up {n} to #{rank}"), `url = /leaderboard`. Idempotent per (match,user) via a push ledger mirroring `result_email_log` (a `channel` discriminator or a sibling `result_push_log`), so re-running `sync-matches` never re-pushes. Reuses `captureRankSnapshot` + `computeRankDelta` unchanged.
- **No new cron, no Realtime.** Push is a batch send off the existing `prediction-reminders` (hourly) and `sync-matches` schedules; no `vercel.json` entry is added and no Supabase Realtime channel is involved. The standing-changed push is intentionally tied to the score-recompute moment, the only point a rank delta exists.
- **Locale & copy.** Push titles/bodies resolve from a new `pushNotifications` i18n namespace at `DEFAULT_LOCALE` (the cron has no request locale, consistent with every email dispatcher). The subscribe-toggle label joins the `emailPrefs` toggle copy.

## Risks / Trade-offs

- **DB migration required.** One new table `public.push_subscriptions` (RLS owner-only) plus a push-idempotency ledger discriminator/sibling table. Purely additive; no change to existing scoring/leaderboard tables. Must live under `supabase/migrations/` with a timestamped filename, matching `20260619153757_email_prefs.sql` and `20260618000000_leaderboard_rank_snapshot.sql`.
- **New infrastructure: service worker + VAPID + subscriptions.** This is the app's first service worker and first browser-persistent subscription store. Scope is contained to push (`public/sw.js` does only push + click; it does not cache or intercept fetch, avoiding accidental offline/caching behavior). VAPID keys must be generated and set in prod; with them unset the whole feature stays dormant (no send), so shipping the migration + UI before keys are configured is safe.
- **iOS limitation.** Safari delivers Web Push only when the site is installed to the Home Screen (PWA). The subscribe UI must detect unsupported/permission-denied states and degrade gracefully (toggle stays off, optional hint). Desktop + Android are unaffected. Documented, not blocking.
- **Stale subscriptions.** Browsers expire endpoints; a send to a dead endpoint returns `410`/`404`. Mitigation: the sender reports it and the caller prunes the row (or increments `failure_count` and prunes past a threshold) inline during the cron, so no separate cleanup cron is needed.
- **Idempotency across two channels.** Email and push share the same audience computation but need independent at-most-once ledgers (a player can have been emailed but not pushed, or vice-versa). The discriminator-on-existing-ledger approach (`channel`/`pushed_at`) keeps one table per trigger; a sibling `*_push_log` is the alternative — either is acceptable, decided at implementation, both preserve the existing email idempotency untouched.
- **Cron timing for the hourly match-needed push.** The prediction reminder is local-7am-bucketed, so push fires alongside it once per day per player; the UTC-day ledger prevents the hourly cron from re-pushing. No change to that schedule.
- **Competitive scoring impact: none.** Push reads `v_leaderboard_overall`/`leaderboard_rank_snapshot` only to render copy and writes no scoring/rank/competition data, so leaderboard fairness and the snapshot-before-recompute invariant are untouched.
- **Deliverability vs. email.** Push has no `EMAIL_FROM`/domain dependency, but requires permission + a live subscription + (on iOS) Home-Screen install, so reach is narrower than email — it complements, never replaces, the email crons.
- **New dependency `web-push`.** Small, well-maintained, server-only; isolated behind `lib/notifications/web-push.ts` so the rest of the codebase never imports it directly.
