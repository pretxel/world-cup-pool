## Why

Every email the retention engine sends today is aimed at players who are *still engaged*: the per-match `result` email when a standing changes (`lib/notifications/result-emails.ts`), the daily `prediction_reminder` before a pickable match (`lib/notifications/prediction-reminder-emails.ts`), and the daily `quiz_reminder`. A player who has stopped picking gets nothing — the prediction reminder only fires when they have a *today* match they have not predicted, and a churned player who skipped a few days quietly drops out of the loop. análisis.md flags this as medium bet **M11**: a *comeback* email to inactive players ("you have not predicted in 5 days, you are #X") reactivates churned users who today receive no message at all (análisis.md §"Apuestas medianas" M11, §"Catálogo de interacción" D). The full pattern to build it already exists end-to-end — the cron seam (`app/api/cron/*` + `vercel.json`), the renderer/dispatcher/Resend/i18n pattern (`prediction-reminder-emails.ts` + `prediction-reminder-template.ts`), inactivity signal (`predictions.submitted_at`), the standings source (`v_leaderboard_overall`), the per-type opt-out (`profiles.email_prefs` via `lib/email-prefs.ts`), and the run-recording seam (`lib/operations/record-run.ts`) — so this is a medium-effort, medium-impact addition with no new third-party dependency.

## What Changes

- Add a new once-daily cron route `app/api/cron/comeback-emails/route.ts` (registered in `vercel.json`) that dispatches a comeback nudge to inactive players, following the exact shape of `app/api/cron/prediction-reminders/route.ts` (Bearer `CRON_SECRET` auth, `maxDuration`, isolated dispatch wrapped in `recordRun`, JSON summary response).
- Add a server-only dispatcher `lib/notifications/comeback-emails.ts` mirroring `prediction-reminder-emails.ts`: gate on `env.resendApiKey`, page through profiles, compute the inactive-but-still-pickable recipient set, drop opted-out and recently-emailed recipients, resolve each recipient's email via the admin client, render localized copy, batch-send (≤100/batch) via Resend, and stamp a cooldown ledger only for batches the provider accepted.
- Define **inactive** as: the player has at least one past prediction but their most recent `predictions.submitted_at` is older than an inactivity threshold (default 5 days), AND there is at least one confirmed, still-pickable upcoming match they could predict now (so the nudge is actionable, not a dead end). A player who predicted within the threshold is **suppressed** (recently active).
- The email body gives the recipient a reason to return: the count of days since their last pick, their **current rank** and total points from `v_leaderboard_overall` (null/“unranked” when they have no scored row yet), the next pickable matches, and a single call-to-action deep-linking to `/matches?picks=needed` (the existing "needs pick" deep link used by the prediction reminder).
- Add a pure, dependency-free renderer `lib/notifications/comeback-email-template.ts` following `prediction-reminder-template.ts`: email-safe table HTML with a fixed hex palette, an HTML + plain-text part, all copy supplied by the caller, including the RFC 8058 one-click unsubscribe footer link.
- Add a per-recipient cooldown ledger `comeback_email_log (user_id uuid, sent_at timestamptz)` (Supabase migration) so a player is not re-nudged on consecutive days: a recipient is eligible only if they have no `comeback_email_log` row newer than a cooldown window (default 14 days). A row is written only after Resend accepts the batch, giving at-most-once-per-cooldown delivery across idempotent re-runs and crashes.
- Honor `profiles.email_prefs`: add a new `comeback` preference key (default opted-IN, `!== false` semantics) to `lib/email-prefs.ts` and to the `profiles.email_prefs` jsonb default via migration, and drop opted-out recipients before send using the existing `isOptedIn`.
- Reuse the existing per-profile `unsubscribe_token` for the one-click footer link; add a one-click unsubscribe route `app/api/comeback-emails/unsubscribe/route.ts` modeled on `app/api/prediction-reminders/unsubscribe/route.ts` (GET + RFC 8058 POST, sets only the `comeback` key).
- Add a new `comeback_emails` operation kind to `lib/operations/record-run.ts` (and `lib/operations/schedule.ts`) so the run shows up in the operations control room with the same `success`/`partial`/`error` classification.
- Add a `comebackEmail` i18n namespace to `messages/{en,es,fr,de}.json`.
- No-op silently when `RESEND_API_KEY` is unset (mirrors every existing dispatcher).

Non-goals: per-timezone send segmentation (that is M6), the results digest (that is M7), the recap/comic digest (that is M9), push notifications, an in-app re-engagement surface, win-back of users with no upcoming pickable matches (a tournament-over state), or any change to how scores/standings are computed.

## Capabilities

### New Capabilities
- `comeback-email`: a once-daily re-engagement email to inactive players — those with a past pick but no prediction in the last N days who still have a pickable upcoming match — showing days-since-last-pick, their current rank, and the next matches to pick, sent to opted-in recipients via a new cron, rate-limited by a per-recipient cooldown, to reactivate churned users who otherwise receive no message.

## Impact

- **App**: new `app/api/cron/comeback-emails/route.ts` (cron handler, modeled on `prediction-reminders/route.ts`) and `app/api/comeback-emails/unsubscribe/route.ts` (one-click unsubscribe, modeled on `prediction-reminders/unsubscribe/route.ts`). New entry in `vercel.json` `crons`.
- **Lib**: new `lib/notifications/comeback-emails.ts` (server-only dispatcher) and `lib/notifications/comeback-email-template.ts` (pure renderer). Reuses `lib/env` `emailFrom`/`resendApiKey`/`siteUrl`, `lib/supabase/admin`, `lib/i18n` `DEFAULT_LOCALE`/`localePath`, `getTranslations`, `lib/match-utils` `isConfirmedMatch`/`isLocked`, the paginated-load + `resolveEmail` + `withFromName` patterns from `prediction-reminder-emails.ts`, and `email-sender-config.ts` `checkEmailSenderConfig`.
- **Operations**: add `comeback_emails` to `OperationKind`/`OPERATION_KINDS` in `lib/operations/record-run.ts` and to `OPERATION_SCHEDULES` in `lib/operations/schedule.ts`; the cron records each run via `recordRun("comeback_emails", "cron", …)`.
- **i18n**: new `comebackEmail` namespace in `messages/{en,es,fr,de}.json`.
- **Data**: one new service-role-only table (Supabase migration) — `comeback_email_log` (per-recipient cooldown ledger, keyed `(user_id, sent_at)` with an index on `(user_id, sent_at)`), RLS enabled with no policies (service-role only), matching `prediction_reminder_log`/`result_email_log` — plus a new `comeback` key in the `profiles.email_prefs` jsonb default and in `lib/email-prefs.ts` (`EMAIL_PREF_KEYS`, `DEFAULT_EMAIL_PREFS`, `emailPrefsSchema`, `normalizeEmailPrefs`). Existing rows are safe because `isOptedIn` reads a missing key as opted-in, so no destructive backfill is needed. The cooldown table needs no historical backfill (an empty ledger correctly makes every currently-inactive player eligible for the first nudge).
- **Dependency / caveat**: deliverability depends on `EMAIL_FROM` being a Resend verified-domain sender in production (`lib/env.ts:46` otherwise falls back to the sandbox `World Cup Pools <onboarding@resend.dev>`, which only reaches the account owner) — análisis.md quick win **QW1**, a prerequisite for every email feature; in this repo `EMAIL_FROM=no-reply@edselserrano.com` with a verified domain is already configured in prod. With `RESEND_API_KEY` unset the dispatch no-ops.
